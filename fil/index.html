Tungkol sa iyong pampublikong IP at lokasyon

Ang iyong pampublikong IP ay ang iyong panlabas na address sa internet. Makikita ito ng mga website at app at mahihinuha ang iyong tiNATayang lokasyon, ISP at timezone. Ang lokasyon sa itaas ay nagmula sa geo lookup ng iyong pampublikong IP, kaya ito ay isang pagtatantya sa antas ng lungsod, hindi isang address ng kalye. Kung ikaw ay nasa isang VPN, dapat nitong IPakita ang lokasyon ng node na iyong pinili, hindi ang iyong tunay na lungsod.

Ano ang DNS leak, at bakit hindi ito ganap na masubukan ng isang browser lamang

Niresolba ng DNS ang mga pangalan ng domain sa mga IP. Kung ikaw ay nasa isang VPN ngunit ang mga query sa DNS ay dumaan pa rin sa iyong lokal na ISP resolver, ang mga site na binibisita mo ay nakalantad sa iyong ISP; iyon ay isang DNS leak. Sa totoo lang: ang isang tunay na pagsubok sa pagtagas ng DNS ay nangangailangan ng bahagi sa panig ng server (ang iyong devICE ay nagresolba ng isang NATatanging subdomain at makikita ng pagsubok kung aling resolver ang nagtanong), na hindi ganap na magagawa ng isang browser lamang. Ang praktikal na diskarte ay ang magpatakbo ng nakalaang DNS leak test site kapag nakakonekta ka na.

Ano ang isang pagtagas ng WebRTC, at kung paano ito lumalampas sa isang VPN

Ang WebRTC ay ang built-in na real-time na API ng komunikasyon ng browser. Ang isang web page ay maaaring lumikha ng isang RTCPeerConnection sa JavaScrIPt; para mag-set up ng peer-to-peer, kinukuha ng browser ang iyong network address sa pamamagitan ng ICE, kasama ang iyong lokal na LAN IP at ang pampublikong IP na IPinapakita ng isang STUN server, at mababasa ng page ang mga kandidatong iyon. Iyon ay isang pagtagas ng WebRTC, at tahimik itong nangyayari nang walang prompt ng pahintulot. Mula noong 2020, ang Chrome at Edge ay gumagamit ng MDNS upang itago ang lokal na IP sa likod ng isang random na .local hostname, ngunit ang pampublikong IP ay maaari pa ring tumagas sa pamamagitan ng STUN reflexive na mga kandidato kahit na ang isang VPN ay naka-on. Ang pagsubok sa itaas ay eksaktong nagpapakita kung aling mga address ang kasalukuyang inilalantad ng iyong browser.

Paano basahin ang mga resultang ito: kung ano ang dapat mong makita sa isang VPN

Ideal na estado: IPinapakita ng pampublikong IP at lokasyon ang iyong napiling lokasyon ng VPN node, hindi inilalantad ng WebRTC ang iyong tunay na pampublikong IP, at ang timezone ng browser ay tumutugma din sa rehiyon ng node. Kung ang pampublikong IP pa rin ang iyong tunay na lokasyon, o inilantad ng WebRTC ang iyong tunay na pampublikong IP, hindi talaga dumadaan ang trapiko sa VPN o mayroong pagtagas; subukan ang isa pang node o protocol at muling subukan. Kung hindi mo pa ito na-install at gusto ng isang kliyente na may mas matatag na koneksyon, magtungo sa ang opisyal na pahina ng pag-download ng LetsVPN.

FAQ

Pagkatapos kumonekta sa isang VPN, makikita pa ba ng mga site ang aking tunay na IP? Maaari bang tumagas ang DNS?

Karaniwan, kapag ang isang VPN ay konektado sa mga site, makikita ang node IP, hindi ang iyong tunay na IP. Ngunit sa totoo lang: ang isang independiyenteng pagsusuri ay nag-flag ng panganib sa pagtagas ng DNS sa LetsVPN at na ang IP ay hindi palaging nagbabago pagkatapos lumIPat ng mga server - iyon ay isang solong mas lumang pagsusuri sa komunidad, na hindi kinumpirma ng vendor o ng pangalawang pinagmulan. Kung mahalaga sa iyo ang privacy, ang praktikal na hakbang ay ang magpatakbo ng IP/DNS leak test tulad nito pagkatapos kumonekta at kumpirmahin na hindi nalantad ang iyong totoong IP at DNS; kung oo, lumIPat ng node o protocol at i-verify muli.

Pagkatapos kumonekta, saan IPinapakita ang aking IP? ItiNATago ba talaga nito ang aking lokasyon?

Kapag nakakonekta na, ang iyong panlabas na IP ay ang lokasyon ng node na iyong pinili - pumili ng Hong Kong node at magpapakita ka ng lokasyon sa Hong Kong; iyan ay kung paano itiNATago ng VPN ang iyong tunay na lokasyon. Maaari kang lumIPat ng mga node ng rehiyon sa kliyente upang baguhin ang iyong panlabas na IP. Tandaan na ang isang independiyenteng pagsusuri ay nag-ulat na ang IP ay hindi palaging nagbabago pagkatapos lumIPat ng mga server at nakita ang DNS leak na panganib (single, mas lumang pagsusuri, hindi multi-source na nakumpirma), kaya pagkatapos kumonekta ay matalinong i-verify gamit ang isang IP test na ang iyong lokasyon ay aktwal na nagbago.

Pagkatapos kumonekta, hindi nagbago ang aking IP at nagpapakita pa rin ng lokal ang mga paghahanap - nabigo ba ito?

Oo, ang isang hindi nabagong IP ay karaniwang nangangahulugan na ang trapiko ay hindi talaga dumadaan sa VPN. Iniulat ng komunidad na ang IP ng LetsVPN ay hindi palaging nagbabago pagkatapos lumIPat ng mga server, na eksakto sa kasong ito. Mga pag-aayos: muling kumonekta sa ibang node ng rehiyon, pagkatapos ay sumubok ng isa pang protocol; i-clear ang cache at mag-log in muli, i-update sa pinakabagong bersyon o muling i-install; siguraduhing walang ibang VPN/proxy ang kumukuha ng trapiko. Pagkatapos kumonekta muli, tingnan muli ang iyong IP dito - mabibilang lamang ito bilang tagumpay kapag IPinakita nito ang iyong napiling bansa. Sa Android, siguraduhin din na hindi ito piNATay ng system sa background.

Ano ang isang pagtagas ng WebRTC, at bakit maaari pa ring tumagas ang aking tunay na IP sa isang VPN?

Ang WebRTC ay ang built-in na peer-to-peer API ng browser. Sa ilang linya lamang ng JavaScrIPt, maaaring lumikha ang isang pahina ng RTCPeerConnection, at pagkatapos ay kumukuha ang browser ng mga address ng network sa pamamagitan ng ICE; ang pampublikong IP na makikita ng isang STUN server ay maaaring maging iyong tunay na IP - kahit na sa isang VPN - at binabasa ito ng pahina nang tahimik nang walang prompt ng pahintulot. Mula noong 2020, tinakpan ng Chrome/Edge ang lokal na IP sa pamamagitan ng MDNS, ngunit maaari pa ring tumagas ang pampublikong IP. Gamitin ang pagsubok sa WebRTC sa itaas upang makita kung aling mga address ang kasalukuyang inilalantad ng iyong browser.
