ການທົດສອບການຮົ່ວໄຫຼ IP / DNS ແລະຕົວເບິ່ງພູມສາດ

ໂດຍ 李航 · ບັນນາທິການທົບທວນ · ຈັດພີມມາ 2025-04-08 · ອັບເດດ 2026-06-30

ເຄື່ອງມືດ້ານລູກຄ້າທີ່ບໍລິສຸດ: ຢູ່ໃນຕົວທ່ອງເວັບຂອງທ່ານມັນອ່ານແລະສະແດງ IP ສາທາລະນະທີ່ທ່ານເປີດເຜີຍກັບອິນເຕີເນັດ, ສະຖານທີ່ຕັ້ງພູມສາດແລະເວລາຂອງທ່ານ, ບວກກັບວ່າ WebRTC ຂອງຕົວທ່ອງເວັບຂອງທ່ານຮົ່ວ IP ທ້ອງຖິ່ນຫຼືທີ່ແທ້ຈິງຂອງທ່ານໄປຫາຫນ້າເວັບຕ່າງໆ. ທຸກໆການກວດສອບແລ່ນຢູ່ໃນທ້ອງຖິ່ນ, ບໍ່ມີຫຍັງຖືກອັບໂຫລດ. ຫຼັງຈາກເຊື່ອມຕໍ່ VPN ເຊັ່ນ LetsVPN, ເປີດໃຊ້ມັນອີກຄັ້ງເພື່ອຢືນຢັນສະຖານທີ່ເບິ່ງ node IP, ບໍ່ແມ່ນອັນແທ້ຈິງຂອງເຈົ້າ.

ເຄື່ອງມືນີ້ເຮັດວຽກທັງຫມົດໃນຕົວທ່ອງເວັບຂອງທ່ານແລະເກັບກໍາຫຼືອັບໂຫລດບໍ່ມີຫຍັງ.

ກ່ຽວກັບ IP ແລະສະຖານທີ່ສາທາລະນະຂອງທ່ານ

IP ສາທາລະນະຂອງທ່ານແມ່ນທີ່ຢູ່ພາຍນອກຂອງທ່ານໃນອິນເຕີເນັດ. ເວັບໄຊທ໌ ແລະແອັບຯສາມາດເບິ່ງເຫັນມັນໄດ້ ແລະສະຫຼຸບສະຖານທີ່ໂດຍປະມານຂອງເຈົ້າ, ISP ແລະເຂດເວລາ. ສະຖານທີ່ຂ້າງເທິງແມ່ນມາຈາກການຊອກຫາທາງພູມສາດຂອງ IP ສາທາລະນະຂອງທ່ານ, ສະນັ້ນມັນເປັນການປະເມີນລະດັບເມືອງ, ບໍ່ແມ່ນທີ່ຢູ່ຕາມຖະຫນົນ. ຖ້າທ່ານຢູ່ໃນ VPN, ມັນຄວນຈະສະແດງສະຖານທີ່ຂອງ node ທີ່ທ່ານເລືອກ, ບໍ່ແມ່ນເມືອງທີ່ແທ້ຈິງຂອງທ່ານ.

DNS leak ແມ່ນຫຍັງ, ແລະເປັນຫຍັງຕົວທ່ອງເວັບດຽວບໍ່ສາມາດທົດສອບມັນໄດ້ຢ່າງເຕັມສ່ວນ

DNS ແກ້ໄຂຊື່ໂດເມນເຂົ້າໄປໃນ IPs. ຖ້າທ່ານຢູ່ໃນ VPN ແຕ່ການສອບຖາມ DNS ຍັງຄົງຜ່ານຕົວແກ້ໄຂ ISP ທ້ອງຖິ່ນຂອງທ່ານ, ສະຖານທີ່ທີ່ທ່ານໄປຢ້ຽມຢາມຈະຖືກເປີດເຜີຍກັບ ISP ຂອງທ່ານ; ນັ້ນແມ່ນການຮົ່ວໄຫລ DNS. ເພື່ອຄວາມຊື່ສັດ: ການທົດສອບການຮົ່ວໄຫລ DNS ທີ່ແທ້ຈິງຕ້ອງການອົງປະກອບຂ້າງເຊີຟເວີ (ອຸປະກອນຂອງທ່ານແກ້ໄຂໂດເມນຍ່ອຍທີ່ເປັນເອກະລັກແລະການທົດສອບເຫັນວ່າຕົວແກ້ໄຂທີ່ຖາມ), ເຊິ່ງຕົວທ່ອງເວັບດຽວບໍ່ສາມາດເຮັດໄດ້ຢ່າງເຕັມສ່ວນ. ວິທີການປະຕິບັດແມ່ນເພື່ອດໍາເນີນການສະຖານທີ່ທົດສອບການຮົ່ວໄຫລ DNS ສະເພາະເມື່ອທ່ານເຊື່ອມຕໍ່.

ການຮົ່ວໄຫຼ WebRTC ແມ່ນຫຍັງ, ແລະມັນຂ້າມ VPN ແນວໃດ

WebRTC ແມ່ນ API ການສື່ສານໃນເວລາຈິງຂອງຕົວທ່ອງເວັບ. ຫນ້າເວັບສາມາດສ້າງ RTCPeerConnection ໃນ JavaScrIPt; ເພື່ອຕັ້ງຄ່າ peer-to-peer ຕົວທ່ອງເວັບລວບລວມທີ່ຢູ່ເຄືອຂ່າຍຂອງທ່ານຜ່ານ ICE, ລວມທັງ IP LAN ທ້ອງຖິ່ນຂອງທ່ານແລະ IP ສາທາລະນະທີ່ສະທ້ອນໂດຍເຄື່ອງແມ່ຂ່າຍ STUN, ແລະຫນ້າສາມາດອ່ານຜູ້ສະຫມັກເຫຼົ່ານັ້ນ. ນັ້ນແມ່ນການຮົ່ວໄຫລຂອງ WebRTC, ແລະມັນເກີດຂຶ້ນຢ່າງງຽບໆໂດຍບໍ່ມີການເຕືອນການອະນຸຍາດ. ນັບຕັ້ງແຕ່ປີ 2020 Chrome ແລະ Edge ໃຊ້ MDNS ເພື່ອປິດບັງ IP ທ້ອງຖິ່ນທີ່ຢູ່ເບື້ອງຫຼັງ .local hostname ແບບສຸ່ມ, ແຕ່ IP ສາທາລະນະຍັງສາມາດຮົ່ວຜ່ານຜູ້ສະຫມັກ STUN reflexive ເຖິງແມ່ນວ່າໃນຂະນະທີ່ VPN ເປີດຢູ່. ການທົດສອບຂ້າງເທິງສະແດງໃຫ້ເຫັນແນ່ນອນວ່າທີ່ຢູ່ຂອງຕົວທ່ອງເວັບຂອງທ່ານເປີດເຜີຍໃນປັດຈຸບັນ.

ວິທີການອ່ານຜົນໄດ້ຮັບເຫຼົ່ານີ້: ສິ່ງທີ່ທ່ານຄວນເຫັນໃນ VPN

ສະຖານະທີ່ເຫມາະສົມ: IP ສາທາລະນະແລະສະຖານທີ່ສະແດງໃຫ້ເຫັນສະຖານທີ່ຂອງ node VPN ທີ່ທ່ານເລືອກ, WebRTC ບໍ່ໄດ້ເປີດເຜີຍ IP ສາທາລະນະທີ່ແທ້ຈິງຂອງທ່ານ, ແລະເຂດເວລາຂອງຕົວທ່ອງເວັບແມ່ນກົງກັບພາກພື້ນ node ເຊັ່ນກັນ. ຖ້າ IP ສາທາລະນະຍັງເປັນສະຖານທີ່ທີ່ແທ້ຈິງຂອງເຈົ້າ, ຫຼື WebRTC ເປີດເຜີຍ IP ສາທາລະນະທີ່ແທ້ຈິງຂອງເຈົ້າ, ການຈະລາຈອນບໍ່ໄດ້ຜ່ານ VPN ແທ້ໆຫຼືມີການຮົ່ວໄຫຼ; ລອງ node ຫຼື protocol ອື່ນ ແລະທົດສອບອີກຄັ້ງ. ຖ້າທ່ານຍັງບໍ່ໄດ້ຕິດຕັ້ງມັນເທື່ອແລະຕ້ອງການລູກຄ້າທີ່ມີການເຊື່ອມຕໍ່ທີ່ຫມັ້ນຄົງ, ໃຫ້ໄປທີ່ ຫນ້າດາວໂຫລດ LetsVPN ຢ່າງເປັນທາງການ.

ລາຍການທີ່ໝາຍ ★ອ້າງອີງສະຖິຕິສາທາລະນະ; ສ່ວນທີ່ເຫຼືອແມ່ນການຄາດຄະເນຂອງເວັບໄຊທ໌ນີ້ຈາກຂໍ້ມູນສາທາລະນະ, ບໍ່ແມ່ນການວັດແທກຂອງບຸກຄົນທໍາອິດ:

  • ★ ຕໍ່ Security.org ແລະອື່ນໆ, ປະມານ 22-25% ຂອງຜູ້ໃຊ້ອິນເຕີເນັດທີ່ມີອາຍຸ 16+ ໄດ້ໃຊ້ VPN ເປັນປະຈໍາຕະຫຼອດ 2022-2025; ຕະຫຼາດກໍາລັງເຕີບໃຫຍ່.
  • ★ ໃນປີ 2025 ມີຜູ້ໃຊ້ VPN ປະມານ 1.75 ຕື້ຄົນ, ປະມານໜຶ່ງສ່ວນສາມຂອງຜູ້ໃຊ້ອິນເຕີເນັດທັງໝົດ.
  • ★ ການແຜ່ກະຈາຍໃນພາກພື້ນແມ່ນກວ້າງ: ປະມານ 55% ໃນອິນໂດເນເຊຍ, 43% ໃນປະເທດອິນເດຍ, ແລະ 38% ແຕ່ລະຄົນໃນປະເທດໄທແລະມາເລເຊຍ; ອາຊີຕາເວັນອອກສ່ຽງໃຕ້ ແມ່ນໜຶ່ງໃນພາກພື້ນທີ່ມີຜູ້ລ້ຽງລູກສູງສຸດ.
  • ຕົວທ່ອງເວັບ desktop ຕົ້ນຕໍ (Chrome / Edge / Firefox / Safari) ທັງຫມົດເຮັດໃຫ້ WebRTC ເປັນຄ່າເລີ່ມຕົ້ນ; ຈາກນີ້ພວກເຮົາຄາດຄະເນວ່າໃນປີ 2026 ສ່ວນແບ່ງທີ່ສໍາຄັນຂອງຜູ້ໃຊ້ VPN ທີ່ບໍ່ໄດ້ປິດການໃຊ້ງານ WebRTC ຍັງຄົງມີຄວາມສ່ຽງຕໍ່ການຮົ່ວໄຫຼຂອງ IP ສາທາລະນະໂດຍຜ່ານການສະທ້ອນ STUN (ຂອບເຂດຄາດຄະເນ, ບໍ່ໄດ້ວັດແທກ).

ພື້ນຖານ: ★ລາຍການອ້າງເຖິງສະຖິຕິສາທາລະນະ (ແຫຼ່ງຂໍ້ມູນຂ້າງລຸ່ມນີ້); ສ່ວນທີ່ເຫຼືອແມ່ນການຄາດຄະເນຂອງເວັບໄຊທ໌ນີ້ຈາກຂໍ້ມູນສາທາລະນະບວກກັບຄວາມຈິງທີ່ວ່າ WebRTC ເປີດໃຊ້ໂດຍຄ່າເລີ່ມຕົ້ນ, ປ້າຍຊື່ "ການຄາດຄະເນ, ບໍ່ແມ່ນບຸກຄົນທໍາອິດ" ແລະບໍ່ໄດ້ນໍາສະເຫນີເປັນຂໍ້ມູນທີ່ວັດແທກ.

ຫຼັງຈາກການເຊື່ອມຕໍ່ VPN, ເວັບໄຊທ໌ຍັງສາມາດເຫັນ IP ທີ່ແທ້ຈິງຂອງຂ້ອຍໄດ້ບໍ? DNS ສາມາດຮົ່ວໄດ້ບໍ?

ໂດຍປົກກະຕິ, ເມື່ອ VPN ເຊື່ອມຕໍ່ເວັບໄຊທ໌ຈະເຫັນ node IP, ບໍ່ແມ່ນ IP ທີ່ແທ້ຈິງຂອງເຈົ້າ. ແຕ່ເພື່ອຄວາມຊື່ສັດ: ການທົບທວນຄືນເອກະລາດໄດ້ລາຍງານຄວາມສ່ຽງຕໍ່ການຮົ່ວໄຫລຂອງ DNS ໃນ LetsVPN ແລະວ່າ IP ບໍ່ໄດ້ປ່ຽນແປງສະເຫມີຫຼັງຈາກການປ່ຽນເຄື່ອງແມ່ຂ່າຍ - ນັ້ນແມ່ນການທົບທວນຊຸມຊົນທີ່ເກົ່າກວ່າ, ບໍ່ໄດ້ຮັບການຢືນຢັນຈາກຜູ້ຂາຍຫຼືແຫຼ່ງທີສອງ. ຖ້າຄວາມເປັນສ່ວນຕົວສໍາຄັນຕໍ່ກັບທ່ານ, ການປະຕິບັດຕົວຈິງແມ່ນເພື່ອດໍາເນີນການທົດສອບການຮົ່ວໄຫລ IP / DNS ເຊັ່ນນີ້ຫຼັງຈາກການເຊື່ອມຕໍ່ແລະຢືນຢັນວ່າ IP ທີ່ແທ້ຈິງຂອງທ່ານແລະ DNS ບໍ່ໄດ້ຖືກເປີດເຜີຍ; ຖ້າພວກເຂົາເປັນ, ສະຫຼັບ node ຫຼື protocol ແລະກວດສອບອີກເທື່ອຫນຶ່ງ.

ຫຼັງຈາກການເຊື່ອມຕໍ່, IP ຂອງຂ້ອຍສະແດງຢູ່ໃສ? ມັນເຊື່ອງສະຖານທີ່ຂອງຂ້ອຍແທ້ໆບໍ?

ເມື່ອເຊື່ອມຕໍ່ແລ້ວ, IP ພາຍນອກຂອງທ່ານແມ່ນສະຖານທີ່ຂອງ node ທີ່ທ່ານເລືອກ - ເລືອກ node ຮ່ອງກົງແລະທ່ານສະແດງສະຖານທີ່ຮ່ອງກົງ; ນັ້ນແມ່ນວິທີທີ່ VPN ເຊື່ອງສະຖານທີ່ທີ່ແທ້ຈິງຂອງເຈົ້າ. ທ່ານ​ສາ​ມາດ​ສະ​ຫຼັບ nodes ພາກ​ພື້ນ​ໃນ​ລູກ​ຄ້າ​ທີ່​ຈະ​ປ່ຽນ IP ພາຍ​ນອກ​ຂອງ​ທ່ານ​. ໃຫ້ສັງເກດວ່າການທົບທວນເອກະລາດໄດ້ລາຍງານວ່າ IP ບໍ່ໄດ້ປ່ຽນແປງສະເຫມີຫຼັງຈາກການປ່ຽນເຄື່ອງແມ່ຂ່າຍແລະເຫັນວ່າຄວາມສ່ຽງຕໍ່ການຮົ່ວໄຫລຂອງ DNS (ການທົບທວນທີ່ເກົ່າແກ່, ບໍ່ແມ່ນການຢືນຢັນຫຼາຍແຫຼ່ງ), ດັ່ງນັ້ນຫຼັງຈາກເຊື່ອມຕໍ່ມັນສະຫລາດທີ່ຈະກວດສອບດ້ວຍການທົດສອບ IP ວ່າສະຖານທີ່ຂອງເຈົ້າມີການປ່ຽນແປງແທ້ໆ.

ຫຼັງຈາກການເຊື່ອມຕໍ່, IP ຂອງຂ້ອຍບໍ່ປ່ຽນແປງແລະການຊອກຫາຍັງຄົງສະແດງຢູ່ໃນທ້ອງຖິ່ນ - ມັນລົ້ມເຫລວບໍ?

ແມ່ນແລ້ວ, IP ທີ່ບໍ່ປ່ຽນແປງໂດຍປົກກະຕິຫມາຍຄວາມວ່າການຈະລາຈອນບໍ່ໄດ້ຜ່ານ VPN. ຊຸມຊົນໄດ້ລາຍງານວ່າ IP ຂອງ LetsVPN ບໍ່ປ່ຽນແປງສະເຫມີຫຼັງຈາກການປ່ຽນເຄື່ອງແມ່ຂ່າຍ, ເຊິ່ງແນ່ນອນແມ່ນກໍລະນີນີ້. ການ​ແກ້​ໄຂ​: ເຊື່ອມ​ຕໍ່​ກັບ node ພາກ​ພື້ນ​ທີ່​ແຕກ​ຕ່າງ​ກັນ​, ຫຼັງ​ຈາກ​ນັ້ນ​ພະ​ຍາ​ຍາມ​ອະ​ນຸ​ຍາດ​ອື່ນ​; ລຶບ cache ແລະເຂົ້າສູ່ລະບົບອີກເທື່ອຫນຶ່ງ, ອັບເດດເປັນເວີຊັນຫຼ້າສຸດຫຼືຕິດຕັ້ງໃຫມ່; ໃຫ້ແນ່ໃຈວ່າບໍ່ມີ VPN/proxy ອື່ນໄດ້ຮັບການຈະລາຈອນ. ຫຼັງຈາກການເຊື່ອມຕໍ່ຄືນໃຫມ່, ໃຫ້ກວດເບິ່ງ IP ຂອງທ່ານອີກເທື່ອຫນຶ່ງທີ່ນີ້ - ມັນຖືວ່າເປັນຜົນສໍາເລັດພຽງແຕ່ເມື່ອມັນສະແດງໃຫ້ເຫັນປະເທດທີ່ທ່ານເລືອກ. ໃນ Android, ໃຫ້ແນ່ໃຈວ່າລະບົບບໍ່ໄດ້ຂ້າມັນຢູ່ໃນພື້ນຫລັງ.

WebRTC ຮົ່ວໄຫຼແມ່ນຫຍັງ, ແລະເປັນຫຍັງ IP ທີ່ແທ້ຈິງຂອງຂ້ອຍຍັງຮົ່ວຢູ່ໃນ VPN?

WebRTC ແມ່ນ API ຂອງ peer-to-peer ໃນຕົວຂອງຕົວທ່ອງເວັບ. ມີພຽງແຕ່ສອງສາມແຖວຂອງ JavaScrIPt ຫນ້າເວັບສາມາດສ້າງ RTCPeerConnection, ແລະຕົວທ່ອງເວັບຈະລວບລວມທີ່ຢູ່ເຄືອຂ່າຍຜ່ານ ICE; IP ສາທາລະນະທີ່ສະທ້ອນໃຫ້ເຫັນໂດຍເຄື່ອງແມ່ຂ່າຍ STUN ສາມາດເປັນ IP ທີ່ແທ້ຈິງຂອງເຈົ້າ - ເຖິງແມ່ນວ່າຢູ່ໃນ VPN - ແລະຫນ້າເວັບຈະອ່ານມັນຢ່າງງຽບໆໂດຍບໍ່ມີການກະຕຸ້ນເຕືອນ. ຕັ້ງແຕ່ປີ 2020 Chrome/Edge ປິດບັງ IP ທ້ອງຖິ່ນຜ່ານ MDNS, ແຕ່ IP ສາທາລະນະຍັງສາມາດຮົ່ວໄດ້. ໃຊ້ການທົດສອບ WebRTC ຂ້າງເທິງເພື່ອເບິ່ງວ່າທີ່ຢູ່ໃດທີ່ຕົວທ່ອງເວັບຂອງທ່ານເປີດເຜີຍໃນປັດຈຸບັນ.

ນະໂຍບາຍຄວາມເປັນສ່ວນຕົວຂອງ LetsVPN ເວົ້າຫຍັງ - ມັນແມ່ນບໍ່ມີບັນທຶກຫຼືພຽງແຕ່ເວົ້າ?

ດ້ວຍຄວາມຊື່ສັດ, ເອກະສານສາທາລະນະທີ່ມີຢູ່ບໍ່ມີຂໍ້ຄວາມນະໂຍບາຍຄວາມເປັນສ່ວນຕົວຂອງ LetsVPN ຫຼືເນື້ອໃນທີ່ບໍ່ມີບັນທຶກທີ່ສາມາດຢືນຢັນໄດ້, ດັ່ງນັ້ນມັນບໍ່ສາມາດຖືກຮັບຮອງ; ທີ່ຕ້ອງການຂໍ້ຄວາມນະໂຍບາຍຢ່າງເປັນທາງການແລະການກວດສອບພາກສ່ວນທີສາມ. ສໍາລັບສະພາບການ, ການທົບທວນຄືນເອກະລາດໄດ້ຍົກຄວາມກັງວົນກ່ຽວກັບການຮົ່ວໄຫລ DNS (ອັນດຽວ, ເກົ່າ, ບໍ່ແມ່ນແຫຼ່ງຫຼາຍທີ່ໄດ້ຮັບການຢືນຢັນ), ເຊິ່ງນັ່ງຢູ່ກົງກັນຂ້າມກັບການຫ້າມຢ່າງເຂັ້ມງວດ. ການໂທຫາການປະຕິບັດ: ປະຕິບັດມັນເປັນເຄື່ອງມືທີ່ໃຊ້ໃນພາກພື້ນ / ທົ່ວໄປ, ໃຊ້ບໍລິການກວດສອບສໍາລັບວຽກງານທີ່ມີຄວາມອ່ອນໄຫວສູງ, ແລະດໍາເນີນການທົດສອບການຮົ່ວໄຫຼດ້ວຍຕົວເອງຫຼັງຈາກການເຊື່ອມຕໍ່ແທນທີ່ຈະໄວ້ວາງໃຈກັບຄໍາຮ້ອງຂໍການຕະຫຼາດ.
